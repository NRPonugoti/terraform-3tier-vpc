Design Principles

Zero Overlap

Every VPC uses a unique /16 range. All environments can be connected via VPC Peering or Transit Gateway without any CIDR conflicts.

Predictable Addressing

Given any IP address, you can instantly identify the environment, tier, and AZ from the octets alone. No documentation lookup needed.

Scalable by Design

Each VPC uses only ~3% of its address space. Reserved ranges allow adding AZs, tiers, or larger subnets without re-addressing.

Defense in Depth

Three isolation tiers (Public, App, DB) with independent security groups, NACLs, and route tables. Traffic flows in one direction only.

Consistent Across Environments

Same internal subnet layout in Dev, Staging, UAT, and Production. Only the second octet changes, reducing operational mistakes.

Future-Proof

97% of address space is unallocated per VPC. Room for EKS pod CIDRs, VPN ranges, new regions, and acquired company networks.

Public
App

Resilience Features

Application Load Balancer

Spans both AZs, automatically routes around unhealthy instances. Health checks every 30 seconds.

Auto Scaling Groups

App and Web tiers scale across AZs based on CPU/memory. Minimum 2 instances, max configurable.

RDS Multi-AZ

Synchronous standby in AZ-2. Automatic failover in under 60 seconds. No application changes required.

NAT Gateway HA

Deployed in public subnet AZ-1. For full HA, add a second NAT GW in AZ-2 with per-AZ route tables.

Phase 1 — Foundation (Dev VPC)

Goal: Deploy the Dev VPC as the first environment using Terraform. Validate the entire 3-tier architecture works end-to-end before expanding.

Step-by-Step Implementation

Step 1.1 — Set Up Terraform Project

  • Install Terraform >= 1.5 and configure AWS CLI credentials
  • Create project structure: versions.tf, variables.tf, network.tf
  • Define environment variable (second octet: 10 for Dev)
  • Store state in S3 backend with DynamoDB locking

Step 1.2 — Create VPC + Subnets

  • VPC: 10.10.0.0/16 with DNS hostnames enabled
  • Create 6 subnets across 2 AZs using the tiered CIDR scheme
  • Enable auto-assign public IP on public subnets only
  • Tag every resource with Environment=dev and Tier=web|app|db

Step 1.3 — Internet Gateway + NAT Gateway

  • Create and attach IGW to the VPC
  • Allocate Elastic IP for NAT Gateway
  • Deploy NAT Gateway in public-az1

Step 1.4 — Route Tables

  • public-rt: 0.0.0.0/0 → IGW, associate with public subnets
  • app-private-rt: 0.0.0.0/0 → NAT GW, associate with app subnets
  • db-private-rt: local only (no internet route), associate with DB subnets

Step 1.5 — Security Groups

  • web-tier-sg: HTTP/HTTPS from 0.0.0.0/0, SSH from admin IP
  • app-tier-sg: Port 8080 + SSH from web-tier-sg only
  • db-tier-sg: Port 3306 from app-tier-sg only

Step 1.6 — Validate

  • Run terraform plan and terraform apply
  • Launch test EC2 in each tier; verify SSH hop chain works
  • Confirm DB subnet has no internet access
  • Confirm app subnet reaches internet only via NAT
Exit Criteria: All 6 subnets created, 3 route tables verified, security group chaining validated, SSH hop working end-to-end.

Phase 2 — Expand (Staging + UAT)

Goal: Replicate the validated Dev VPC to Staging and UAT environments. Connect all three VPCs via AWS Transit Gateway for cross-environment communication.

Step-by-Step Implementation

Step 2.1 — Parameterize Terraform Module

  • Refactor Dev Terraform code into a reusable module
  • Parameterize the environment octet: var.env_octet (20 = Staging, 30 = UAT)
  • Create staging.tfvars and uat.tfvars with environment-specific values
  • Use Terraform workspaces or separate state files per environment

Step 2.2 — Deploy Staging VPC

  • VPC: 10.20.0.0/16
  • Run terraform apply -var-file=staging.tfvars
  • Verify same subnet layout: 10.20.1-2, 10.20.16-17, 10.20.32-33
  • Validate security groups and route tables

Step 2.3 — Deploy UAT VPC

  • VPC: 10.30.0.0/16
  • Run terraform apply -var-file=uat.tfvars
  • Verify same subnet layout: 10.30.1-2, 10.30.16-17, 10.30.32-33
  • Validate security groups and route tables

Step 2.4 — Create Transit Gateway

  • Create AWS Transit Gateway in the shared region
  • Create TGW attachments for Dev, Staging, and UAT VPCs
  • Update route tables in each VPC to route cross-VPC traffic via TGW:
    10.10.0.0/16 → TGW (from Staging/UAT)
    10.20.0.0/16 → TGW (from Dev/UAT)
    10.30.0.0/16 → TGW (from Dev/Staging)

Step 2.5 — CI/CD Pipeline Integration

  • Set up Terraform CI/CD pipeline (GitHub Actions / GitLab CI / Jenkins)
  • Implement plan on PR, apply on merge to main
  • Add tflint and tfsec for static analysis
  • Enforce PR review before any infrastructure changes
Exit Criteria: 3 VPCs running with identical layouts, Transit Gateway connecting them, CI/CD pipeline deploying Terraform automatically.

Phase 3 — Production Deployment

Goal: Deploy the Production VPC with full high availability, auto scaling, multi-AZ database, and load balancing. This is the most critical phase and requires change management approval.

Step-by-Step Implementation

Step 3.1 — Deploy Production VPC

  • VPC: 10.40.0.0/16
  • Run terraform apply -var-file=prod.tfvars
  • Verify subnet layout: 10.40.1-2, 10.40.16-17, 10.40.32-33
  • Attach to existing Transit Gateway

Step 3.2 — Application Load Balancer

  • Create internet-facing ALB in both public subnets
  • Configure HTTP (80) and HTTPS (443) listeners
  • Create target groups with health checks (path: /health, interval: 30s)
  • Attach ACM SSL certificate for HTTPS

Step 3.3 — Auto Scaling Groups

  • Create launch templates for web and app instances
  • Web ASG: min=2, max=6, desired=2 across both public subnets
  • App ASG: min=2, max=8, desired=2 across both app subnets
  • Scaling policies: target CPU at 60%, scale-out cooldown 300s

Step 3.4 — RDS Multi-AZ Database

  • Create DB subnet group with both DB subnets
  • Deploy RDS MySQL/PostgreSQL with Multi-AZ enabled
  • Instance class: db.r6g.large (production-grade)
  • Enable automated backups (retention: 7 days), encryption at rest (KMS)
  • Enable Performance Insights and Enhanced Monitoring

Step 3.5 — NAT Gateway HA

  • Deploy a second NAT Gateway in public-az2
  • Split app-private-rt into per-AZ route tables:
    app-private-rt-az1 → NAT GW AZ-1
    app-private-rt-az2 → NAT GW AZ-2
  • Ensures outbound traffic survives a single AZ failure

Step 3.6 — DNS & SSL

  • Create Route 53 hosted zone for the domain
  • Create alias record pointing to ALB DNS
  • Request ACM certificate with DNS validation
  • Add HTTP → HTTPS redirect rule on ALB
Exit Criteria: Production VPC live, ALB serving HTTPS, ASGs scaling, RDS Multi-AZ with backups, dual NAT GWs for full HA.

Phase 4 — Harden & Optimize

Goal: Add security monitoring, compliance guardrails, logging, and cost optimization across all 4 environments to meet enterprise production standards.

Security & Compliance

Step 4.1 — VPC Flow Logs

  • Enable VPC Flow Logs on all 4 VPCs
  • Send to CloudWatch Log Group with 30-day retention
  • Also archive to S3 for long-term audit (90 days+)
  • Create CloudWatch metric filters for rejected traffic

Step 4.2 — GuardDuty & Security Hub

  • Enable Amazon GuardDuty for threat detection
  • Enable AWS Security Hub with CIS Benchmark standard
  • Configure SNS alerts for high-severity findings
  • Set up automatic remediation for common findings

Step 4.3 — AWS Config Rules

  • vpc-sg-open-only-to-authorized-ports: No open SGs
  • subnet-auto-assign-public-ip-disabled: Private subnets only
  • rds-instance-public-access-check: No public DB access
  • encrypted-volumes: All EBS volumes encrypted

Monitoring & Cost Optimization

Step 4.4 — Centralized Logging

  • Enable AWS CloudTrail in all environments
  • Aggregate logs into a central S3 bucket
  • Set up Athena queries for security investigations
  • Create CloudWatch dashboards for each VPC

Step 4.5 — Network ACLs (Defense in Depth)

  • Add NACLs as a second layer of defense per subnet
  • Public NACLs: Allow 80, 443, 22, ephemeral ports; deny all else
  • App NACLs: Allow from 10.ENV.1-2.0/24 on app port only
  • DB NACLs: Allow from 10.ENV.16-17.0/24 on DB port only

Step 4.6 — Cost Optimization

  • Enable AWS Cost Explorer and set budget alerts
  • Tag enforcement: all resources must have Environment + Tier
  • Review NAT Gateway data charges (consider VPC endpoints for S3/DynamoDB)
  • Right-size instances after 2 weeks of CloudWatch metrics
  • Use Reserved Instances or Savings Plans for production
Exit Criteria: Flow Logs on all VPCs, GuardDuty active, Config Rules enforced, CloudTrail logging, NACLs applied, cost baselines established.
